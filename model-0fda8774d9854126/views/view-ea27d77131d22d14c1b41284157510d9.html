Source-to-meta-model mapping ()
The Source-to-meta-model mapping process is initiated by a request or a scheduled event within the Repository Initialisation & Strategy capability. This process encompasses the assessment of the current state, the establishment of the EA strategic direction and roadmap, the transfer of information and seeding of the repository, and the design of the meta-model, ensuring that all elements align with the overarching architectural strategy. It exists to deliver value by ensuring that the foundational architecture repository is established in a way that supports strategic coherence and compliance with architectural standards, ultimately producing the Initial Architecture Blueprint for Architecture Standards & Compliance and leadership conversation output for Strategic Alignment & Leadership Engagement. <!-- SPEC --> **Lifecycle phases** - Current state assessment: evaluates the existing architecture and identifies gaps. - EA strategic direction and roadmap: defines the strategic objectives and the path to achieve them. - Information transfer and repository seeding: populates the repository with relevant data and artifacts. - Meta-model design: creates a structured representation of the architecture elements and their relationships. Steps: 1. Receive meta-model spec: Process is triggered by receiving a meta-model specification from the Meta-model design / Build sub-capability. 2. Assess current state: EA Analyst evaluates the existing architecture and identifies gaps against the meta-model specification. 3. Determine critical gaps: Review whether critical gaps exist that would prevent proceeding; if critical, process is cancelled; otherwise continue. 4. Process cancelled: Process terminates because critical gaps cannot be resolved in this context. 5. Define strategic direction: EA Strategist defines strategic objectives and a roadmap based on the assessment and the meta-model specification. 6. Transfer and seed repository: Repository Administrator populates the architecture repository with relevant data and artifacts. 7. Design meta-model: Meta-model Designer creates a structured representation of architecture elements and relationships. 8. Create mapping document: Meta-model Designer produces a draft mapping document that maps source elements to meta-model constructs. 9. Validate mapping document: EA Architect reviews the draft for accuracy, consistency, and compliance with standards. 10. Revise mapping document: Meta-model Designer corrects issues identified during validation. 11. Send mapping document: The validated mapping document is dispatched to the downstream Information transfer and repository seeding process. 12. Process complete: The Source-to-meta-model mapping process finishes successfully with the document delivered.
